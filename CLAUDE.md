# Slides Plugin

A skill pack that takes someone from a vague idea to a finished, on-brand, slop-free PowerPoint deck. Five skills: `presentation-craft` (shared knowledge, non-invocable) plus four user-invocable skills ‚Äî `teach-slides`, `narrative`, `build-deck`, `slop-check`.

## Design philosophy

The pack constrains to a brand's *identity*, not to a house style, and leaves *composition* to the model. Guardrails and freedom are separated on purpose ‚Äî changes should preserve that separation:

- **Identity is the constraint.** Colour, type, and grid come from the user's own brand, captured as design tokens (with logo and shape language to follow). Every drawn element must resolve to a token; a mechanical lint enforces it, so nothing goes off-brand.
- **Composition is the craft.** How a slide is arranged is the model's judgement, per deck. We never impose a "slides-skill look" ‚Äî a finished deck should read as *the brand*, not as this tool.
- **Extract identity, not layout.** Faithfully filling a template's placeholders reproduces its layouts *and its design debt*. The `composed`/`freeform` path ‚Äî brand-constrained free composition ‚Äî is where good slides come from; the six fixed placeholder roles are the conservative fallback.
- **No generated imagery, no slop.** Visual meaning is built from structured, on-token elements (the composed primitives and the freeform placer), never AI-generated images; a `Visual:` note defers real photography to a human. An adversarial slop-check guards the prose.
- **The lint is the only rigidity.** On-token colour, type-scale size, on-grid, no overlap, and the element cap are absolute; composition quality is advisory. Constrain what is unsafe; leave the rest free.

The craft behind this lives in `source/skills/presentation-craft/reference/` (`slides.md`, `composition.md`, `design-research.md`).

## Structure

### Source (single source of truth)

- `source/skills/` ‚Äî all skill source files. Edit here, not in the generated output directories.
- `source/commands/` ‚Äî OpenCode slash-command wrappers. Edit here.

### Generated output (committed, do not edit directly)

- `.claude/skills/` ‚Äî Claude Code output. Identity transform; source is emitted as-is.
  - `presentation-craft/` ‚Äî Shared presentation-craft knowledge base. Non-invocable.
  - `teach-slides/` ‚Äî Onboards a user to presentation craft.
  - `narrative/` ‚Äî Shapes a vague idea into a deck story.
  - `build-deck/` ‚Äî Renders an on-brand `.pptx` deck.
  - `slop-check/` ‚Äî Audits a deck for generic, AI-flavoured slop.
- `.opencode/skills/` ‚Äî OpenCode output. Contains all five skills. Claude-only fields stripped, `$ARGUMENTS` and `/slides:` references rewritten.
- `.opencode/commands/` ‚Äî OpenCode slash commands. Four flat `.md` files, each a thin `@.opencode/skills/<name>/SKILL.md` include so users can invoke the four user-invocable skills as `/teach-slides`, `/narrative`, `/build-deck`, `/slop-check`. Do not edit directly.
- `.agents/skills/` ‚Äî Codex output for repo-local discovery. All five skills.
- `plugins/slides/skills/` ‚Äî Codex output for Plugins UI install packaging. Byte-identical to `.agents/skills/` (enforced by a 3-way sandbox test in `builder.test.js`). Do not edit directly.
- `.codex/skills/` ‚Äî Codex cross-harness bridge. Byte-identical to `.agents/skills/` via shared `codexRewrites` by reference. Exists so tools that cross-read `.codex/skills/` (notably Cursor) can discover our skills. Do not edit directly.

All five skills are portable: nothing depends on Claude-only sub-agent or Task tooling, so no provider has an `exclude` list.

### Static manifests (hand-authored, committed)

- `.claude-plugin/plugin.json` + `.claude-plugin/marketplace.json` ‚Äî Claude Code plugin descriptors.
- `plugins/slides/.codex-plugin/plugin.json` ‚Äî Codex plugin manifest.
- `.agents/plugins/marketplace.json` ‚Äî Codex marketplace catalog pointing at `./plugins/slides`.

These four JSON files are NOT regenerated by `npm run build`. Edit them by hand; version bumps must be applied across all four version carriers together (see "Versioning" below).

## Conventions

- Each skill has a SKILL.md with frontmatter (name, description, and, for user-invocable skills, user-invocable, argument-hint, model, effort, context).
- Reference files go in `reference/` subdirectories within the skill that owns them ‚Äî primarily `presentation-craft/reference/`.
- Cross-skill references use the `/slides:skillname` invocation format (Claude syntax; transformer rewrites for other harnesses).
- `presentation-craft` is the shared knowledge base; the four user-invocable skills draw on it rather than duplicating craft guidance.
- All skills must work standalone in every harness.

## Build pipeline

```
npm run build       # regenerate all output directories from source/skills/
npm run check-sync  # verify version parity + outputs match what is committed
npm test            # run transformer, builder, manifest, skill-contract, and check-sync tests
```

Transformer scripts live in `scripts/`. Harness capability details and transform rules are in `HARNESSES.md`.

### Pre-push checklist

Full version-drift coverage requires BOTH commands to pass before push:

- `npm run check-sync` ‚Äî catches structural drift of generated output directories AND version drift via a pre-build parity assertion across all four version carriers listed in `scripts/transformers/version-carriers.js`.
- `npm test` ‚Äî cross-checks the same version parity in-process via `manifests.test.js` (reading the same `VERSION_CARRIERS` list), AND verifies the drift-detection mechanism itself by spawning `check-sync.js` with a mismatched version in `check-sync.test.js`.

Neither command alone is a complete gate. There is no CI; the gate is self-enforced.

## Versioning

Version lives in four files that MUST be bumped together:

- `.claude-plugin/plugin.json` (Claude Code plugin)
- `.claude-plugin/marketplace.json` (Claude Code marketplace ‚Äî `plugins[0].version`)
- `plugins/slides/.codex-plugin/plugin.json` (Codex plugin)
- `package.json` (build pipeline)

The canonical list lives in `scripts/transformers/version-carriers.js`. `npm run check-sync` fails if any of the four disagree. `manifests.test.js` enforces the same parity in-process. See the pre-push checklist above.

## Testing

Test each skill standalone before testing the full idea-to-deck flow. See README.md for verification steps.

After any source change, run `npm run build` before committing.

### Testing caveats (accepted risks)

One known risk in the current test suite, judged acceptable for this single-maintainer repo:

- **`check-sync.test.js` mutates a committed file.** The test writes `"99.99.99"` into `plugins/slides/.codex-plugin/plugin.json`, spawns `check-sync.js` synchronously, and restores in a `try/finally`. If Node dies between the write and the restore (vanishingly unlikely ‚Äî `spawnSync` is synchronous), the working tree is dirty. Recovery: `git checkout -- plugins/slides/.codex-plugin/plugin.json`. Concurrency with other writers to the same file is avoided by `--test-concurrency=1` in the `test` script; do not remove that flag without redesigning the test to sandbox the file.
